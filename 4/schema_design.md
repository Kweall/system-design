# Документная модель MongoDB для Pillow API

## Выбор между embedded и references

### Сущности
- Пользователи (`users`)
- Объекты недвижимости (`properties`)
- Просмотры (`viewings`)

### Принятое решение
- **Embedded (вложенные документы) для `viewings`** — внутри каждого `property`
- **References не используются**

### Обоснование

1. **Один объект → несколько просмотров**
   - У одного объекта недвижимости обычно не более 50–100 просмотров
   - Это небольшое количество, вложение не приведёт к превышению лимита 16MB

2. **Частота запросов**
   - `GET /properties/{id}` — самый частый запрос в системе
   - При embedded все просмотры приходят за один запрос к MongoDB
   - При references потребовалось бы два запроса (или `$lookup`)

3. **Обновления**
   - Просмотры обновляются редко (только статус)
   - Обновление вложенного документа через `$set` с позиционным оператором `$` — эффективно

4. **Удаления**
   - При удалении объекта все его просмотры удаляются автоматически
   - Не нужно отдельно чистить коллекцию `viewings`

### Структура документа `properties`

```json
{
  "_id": ObjectId,
  "property_id": 101,
  "title": "Modern Apartment",
  "price": 250000,
  "city": "Moscow",
  "rooms": 2,
  "owner": {
    "user_id": 1,
    "username": "alice"
  },
  "viewings": [
    {
      "viewing_id": 1001,
      "user_id": 2,
      "scheduled_time": ISODate("..."),
      "status": "scheduled",
      "notes": "...",
      "created_at": ISODate("...")
    }
  ],
  "created_at": ISODate("..."),
  "updated_at": ISODate("...")
}
```

### Индексы для производительности

| Индекс | Назначение |
|--------|-----------|
| `property_id: 1` (unique) | Быстрый поиск по ID |
| `city: 1` | Фильтрация по городу |
| `price: 1` | Фильтрация по цене |
| `city: 1, price: 1` | Составной фильтр (самый частый) |
| `owner.user_id: 1` | Поиск объектов пользователя |
