# Документирование архитектуры в Structurizr

**Вариант 24: Система управления недвижимостью (аналог Zillow)**

## Цель работы
Получить навык в описании архитектуры в стиле Architecture As A Code и проектировании системы «сверху вниз».

## Описание выбранного варианта
Разрабатывается система управления недвижимостью, которая позволяет пользователям добавлять объекты недвижимости, искать их по различным критериям и записываться на просмотры.

### Данные системы
- **Пользователь**: зарегистрированный пользователь системы
- **Объект недвижимости**: квартира, дом, офис и т.д.
- **Просмотр**: запись на осмотр объекта недвижимости

### Функциональные требования (API)
- Создание нового пользователя
- Поиск пользователя по логину
- Поиск пользователя по маске имени и фамилии
- Добавление объекта недвижимости
- Поиск объектов по городу
- Поиск объектов по цене
- Запись на просмотр объекта
- Получение записей на просмотр объекта
- Получение объектов пользователя
- Изменение статуса объекта

## Выполненные задачи

### 1. Определение ролей пользователей и внешних систем

**Роли пользователей:**
- **Гость** - незарегистрированный пользователь, который может только искать объекты недвижимости
- **Зарегистрированный пользователь** - может добавлять объекты, записываться на просмотры, управлять своими объектами
- **Администратор** - управляет системой, имеет доступ к панели администрирования

**Внешние системы:**
- **Email Service** - сервис для отправки email-уведомлений пользователям (подтверждение записи на просмотр, уведомления о статусе объектов)

### 2. Диаграмма System Context (C1)

Диаграмма контекста системы показывает:
- Взаимодействие всех трех ролей пользователей с системой "Pillow"
- Интеграцию с внешним Email-сервисом для отправки уведомлений
- Основные сценарии использования для каждой роли

### 3. Определение контейнеров системы

На основе анализа задач пользователей определены следующие контейнеры:

| Контейнер | Технологии | Ответственность |
|-----------|------------|-----------------|
| **Web Application** | React | Веб-интерфейс для всех категорий пользователей |
| **API Gateway** | Node.js Express | Маршрутизация запросов к микросервисам |
| **User Service** | Python FastAPI | Управление пользователями (регистрация, поиск) |
| **Property Service** | Java Spring Boot | Управление объектами недвижимости |
| **Viewing Service** | Go | Управление записью на просмотры |
| **Database** | PostgreSQL | Централизованное хранение всех данных |

### 4. Взаимодействие между контейнерами

Основные сценарии взаимодействия:
- Маршрутизация всех клиентских запросов через API Gateway
- Разделение бизнес-логики по микросервисам (User, Property, Viewing)
- Взаимодействие между сервисами для проверки данных (Viewing Service проверяет существование пользователя и объекта)
- Единая база данных для всех сервисов

### 5. Диаграмма Container (C2)

Диаграмма контейнеров отображает:
- Все 6 контейнеров системы
- Связи между контейнерами с указанием протоколов (HTTPS/REST, PostgreSQL)
- Технологический стек для каждого контейнера
- Направления потоков данных

### 6. Диаграмма Dynamic для сценария "Запись на просмотр"

Динамическая диаграмма для архитектурно значимого сценария детально показывает последовательность шагов:

1. Пользователь нажимает "Записаться на просмотр" в веб-интерфейсе
2. Web Application отправляет POST-запрос к API Gateway
3. API Gateway маршрутизирует запрос к Viewing Service
4. Viewing Service проверяет существование объекта через Property Service
5. Property Service подтверждает наличие объекта
6. Viewing Service проверяет существование пользователя через User Service
7. User Service подтверждает наличие пользователя
8. Viewing Service сохраняет запись в базе данных
9. База данных подтверждает сохранение
10. Viewing Service возвращает данные через API Gateway
11. API Gateway передает ответ веб-приложению
12. Веб-приложение отображает подтверждение пользователю


## Выводы

В результате была спроектирована архитектура системы управления недвижимостью с использованием подхода "Architecture As A Code". Система построена по микросервисной архитектуре с API Gateway, что обеспечивает масштабируемость и независимость компонентов. Все требования покрыты функциональностью соответствующих микросервисов.

Диаграммы C4-модели позволяют наглядно представить архитектуру на разных уровнях абстракции: от контекста системы до детальной последовательности выполнения конкретного сценария.
